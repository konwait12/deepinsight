# DeepInsight 项目交接文档

更新日期：2026-05-21

这份文档面向下一个接手项目的 AI 或开发者。重点是快速理解项目结构、当前状态、启动方式、安全约束和后续开发边界，不追求穷尽所有实现细节。

## 1. 当前启动状态

本轮已尝试启动前后端：

- 前端已启动成功：`http://localhost:3000/`
- MySQL 已监听：`localhost:3306`
- Redis 已监听：`localhost:6379`
- 后端未成功启动：`localhost:8080` 未监听

后端失败原因不是编译代码错误，而是本地环境缺少数据库凭据。日志关键错误：

```text
Access denied for user 'root'@'localhost' (using password: NO)
```

也就是说当前 `.env.local` 没有提供 `DB_PASSWORD` 或可用 `DB_URL`。后端启动前需要补齐数据库连接环境变量。不要把真实密码写进 Git。

补充：如果直接在普通 shell 里运行 `backend/mvnw.cmd package`，可能会因为当前 shell 使用 Java 8 而失败。后端要求 JDK 21，启动脚本会优先使用 `DEEPINSIGHT_JAVA_HOME` 或 `D:\jdk\java21`。

## 2. 项目定位

DeepInsight 是一个深度学习实验工作台，核心目标是把数据、训练、分析结果、AI 对话材料、上传文件和云端素材统一成可管理、可复用、可追踪的研究资产。

主要功能：

- 系统总览
- 数据管理
- 模型训练
- 可视化分析
- AI 工作室
- 云端中心
- 知识库
- 社区论坛
- 管理后台

当前产品方向不是单纯“训练平台”，而是“实验资产管理 + 可视化分析 + AI 辅助复盘”的一体化应用。

## 6. 环境变量

从 `.env.example` 复制为 `.env.local`，至少补齐：

```text
VITE_API_BASE_URL=/api/v1
DB_HOST=localhost
DB_PORT=3306
DB_NAME=deepinsight
DB_USER=root
DB_PASSWORD=你的本地数据库密码
JWT_SECRET=至少32字节的强随机密钥
SERVER_PORT=8080
DEEPINSIGHT_JAVA_HOME=D:\jdk\java21
```

不要提交：

- `.env.local`
- 真实数据库密码
- JWT secret
- DeepSeek/OpenAI/Gemini 等模型 API Key
- 数据库 dump
- Redis dump
- 运行日志
- 构建产物

模型 API Key 不应写进前端环境变量。应通过后台 AI 配置页写入服务器数据库，接口返回时需要脱敏。

## 7. 前后端连接

前端 Vite 运行在 `3000`，代理配置在 `vite.config.ts`：

- `/api` -> `http://localhost:8080`
- `/uploads` -> `http://localhost:8080`

后端默认运行在 `8080`，由 `SERVER_PORT` 覆盖。

如果前端页面能打开但接口失败，优先检查：

1. `backend.run.log`
2. `DB_PASSWORD` / `DB_URL`
3. MySQL 是否有 `deepinsight` 数据库
4. `JWT_SECRET` 是否为空
5. `localhost:8080` 是否监听

## 8. 数据库与迁移

数据库包含真实用户数据和生成内容，不能随便重建。

默认后端配置：

```text
JPA_DDL_AUTO=validate
```

不要依赖 Hibernate 自动 `update` 改表。实体变更必须配套迁移 SQL。

安全迁移流程：

1. `npm run db:backup`
2. `npm run db:rebuild-copy`
3. 在 `backend/src/main/resources/db/` 新建日期命名迁移。
4. 先对复制库执行迁移。
5. 对比关键表行数。
6. 再应用到真实 `deepinsight` 库。

已有关键迁移包括：

- `2026-05-18-ai-workspace.sql`
- `2026-05-18-visual-analysis-saved-records.sql`
- `2026-05-17-visual-analysis-matrix.sql`
- `2026-05-17-bcrypt-only-users.sql`

### 8.1 MySQL 持久化数据库架构

MySQL 是项目的事实来源，负责用户、权限、资产、训练记录、可视化结果、知识内容和云端文件索引。后端默认使用 JPA 仓库读取业务实体，部分新功能也通过 `JdbcTemplate` 创建或补齐表结构，但正式变更仍应沉淀为 `backend/src/main/resources/db/` 迁移 SQL。

核心表按业务域理解：

- 身份与安全：`users` 保存账号、角色、BCrypt 密码哈希；`ai_configs` 保存服务端 AI 模型配置，API Key 只能在后端持久化和脱敏返回。
- 数据与训练：`datasets`、`training_jobs`、`training_steps` 记录数据集元信息、训练任务和训练过程步骤，数据归属分别通过 `uploaded_by`、`created_by` 等字段隔离。
- 实验日志：`experiment_runs` 关联用户实验运行；`scalar_logs`、`image_logs`、`audio_logs`、`text_logs`、`histogram_data`、`embedding_data`、`pr_curve_data`、`roc_curve_data`、`hparam_data`、`profiler_data` 保存可视化分析需要的多模态日志。
- 可视化分析：`visual_analysis_batches`、`visual_analysis_results`、`visual_ai_panels` 保存矩阵分析批次、单项结果和 AI 洞察面板；`visual_saved_analysis_records`、`visual_saved_views` 保存用户收藏的分析记录和视图。
- 云端工作区：`ai_workspace_folders`、`ai_workspace_items` 是服务器数据库里的用户云端空间。文件夹、素材、上传文件、保存结果都必须带 `user_id`；移动、重命名、删除、下载前都要验证归属。
- 知识库与社区：`knowledge_nodes`、`knowledge_docs`、`knowledge_articles`、`forum_posts`、`forum_comments` 保存知识图谱、文档、文章、帖子和评论；官方内容通过 `is_official` 或后台来源字段区分，普通页面不得修改官方资产。
- 模型资产：`model_registry`、`model_articles` 保存模型目录和模型文章。官方模型 `is_official=1`、`created_by=NULL`，只能由 admin 后台管理；普通用户创建的模型必须带创建者归属。

设计边界：

- 所有用户私有数据查询必须显式带当前登录用户 ID，不要只靠前端隐藏。
- 官方资产默认只读；即使当前用户是 admin，在普通用户页面也不能绕过后台管理入口修改官方数据。
- 上传文件可以存路径、URL、BLOB 或 `payload_json`，但接口层必须防路径穿越、危险 MIME、超大文件和越权下载。
- `JPA_DDL_AUTO=validate` 是安全默认值。新增字段或表时，先补迁移 SQL，再改实体和服务。

### 8.2 Redis 运行态数据库架构

Redis 不是项目事实来源，主要承担 AI 会话、短期记忆和在线状态等运行态数据。`spring.data.redis.repositories.enabled=false`，后端通过 `RedisTemplate<String, Object>` 直接读写，序列化配置在 `RedisConfig.java`。

当前键空间：

- `chat:conv:{id}`：AI 对话元数据，包含标题、用户 ID、创建和更新时间。
- `chat:session:{convId}`：对话消息列表，按会话 ID 保存最近历史。
- `user:convs:{userId}`：用户拥有的会话 ID 集合，用于会话列表。
- `user:mem:{userId}`：AI 短期记忆列表。
- `chat:conv:seq`：Redis 自增对话 ID。
- `user:online:{userId}`：在线状态，当前 TTL 为 30 分钟。

TTL 与降级策略：

- AI 对话、消息和记忆默认保留 7 天；在线状态保留 30 分钟。
- Redis 宕机应只影响会话历史、短期记忆、在线状态等体验层能力，不能破坏 MySQL 中的数据资产。
- 不要在 Redis 长期保存明文 API Key、数据库密码、JWT secret 或可恢复的敏感文件内容。
- 如果未来需要“长期 AI 对话归档”，应新增 MySQL 表或对象存储索引，不要单纯拉长 Redis TTL。

## 9. 核心功能状态

系统总览：

- 已做成工作台入口。
- 聚合训练、数据、云端、最近运行和安全状态。

数据管理：

- 不只管理数据集。
- 目标是统一管理所有“数据相关资产”：数据集、训练任务、运行日志、可视化日志、保存视图、矩阵分析结果、云端素材等。
- 官方资产不可由普通用户或 admin 在普通数据管理页修改。
- 官方资产只能在 admin 后台管理。

云端中心：

- 完整页面在 `/cloud`。
- 右上角文件夹入口只做小图标跳转，不再弹小窗。
- 支持服务器数据库中的用户隔离素材、文件夹树、上传、重命名、删除、移动、下载。
- 支持封面墙和列表。
- 支持 AI 工作室选择云端素材作为上下文。

可视化分析：

- 主页面是 `src/views/visualization/VizHub.vue`。
- 支持一对多、多对多矩阵分析。
- 支持保存分析记录和视图到云端。
- 不要随意给可视化主工作区加多余滚动条。

AI 工作室：

- 支持对话、推理层级、材料注入、云端素材导入。
- AI 悬浮助手曾多次调整，注意保留小图标动效和轻量玻璃质感，不要做成廉价强发光。

知识库与论坛：

- 知识库包含文章与图谱。
- 论坛用于社区内容与官方内容展示。
- 论坛背景和玻璃板风格应与全局一致。

后台：

- 用户、模型、AI 配置、知识库、论坛、数据等管理入口在 `/admin`。
- 需要 `ADMIN` 角色。

## 10. 设计与交互约束

用户近期明确偏好：

- 不要“AI 味”重的通用卡片。
- 不要塑料感强的大面积彩色发光。
- 更接近 iOS / Apple 的简洁、克制、干净层级。
- 玻璃/金属质感应靠冷白高光、暗部层次、细边框、低饱和主题色点缀。
- 横向导航栏是重要参考；其他组件应比导航栏更有色彩但更透明。
- 可视化分析页的组件质感是当前相对可接受的参考。
- 鼠标靠近边框光效应该是“以鼠标为中心的隐形圆影响附近组件边缘的一小段”，不能出现中心强光。
- 不合适的组件不要强行加边缘光。
- 文本输入区域不能因为靠近鼠标冒出输入光标或闪烁。

当前主题策略：

- 只保留四个固定皮肤色。
- 不再保留自定义色相/透明度滑条。
- 废弃旧的 `theme-hue`、`theme-saturation`、`theme-lightness`、`theme-glass-opacity` 本地存储。

相关文件：

- `src/stores/theme.store.ts`
- `src/index.css`
- `src/components/common/Header.vue`
- `src/App.vue`

## 11. 安全约束

必须保持：

- JWT 鉴权。
- 密码 BCrypt。
- API Key 不进 Git。
- AI 配置接口仅 admin 可管。
- 下载私有云端素材必须走鉴权接口。
- 云端素材、文件夹、数据资产查询必须带用户隔离。
- 官方资产不能被普通页面改。
- 上传要防路径穿越、危险扩展名、过大文件、错误 MIME。
- CORS 生产环境必须收窄到正式域名。

不要做：

- 不要把 DeepSeek/OpenAI/Gemini Key 写进前端。
- 不要提交 `.env.local`。
- 不要为了“先跑起来”把安全校验删掉。
- 不要让 admin 在普通用户界面绕过官方资产限制。

## 14. 给下一个 AI 的建议

接手后优先顺序：

1. 补齐本地 `.env.local` 的 `DB_PASSWORD`、`JWT_SECRET`，确认后端能在 `8080` 启动。
2. 用真实登录流程检查 `/dashboard`、`/data`、`/cloud`、`/viz`、`/ai`。
3. 优先修复数据/云端/AI 保存链路，不要先做新视觉特效。
4. 做视觉时沿用当前全局 token，不要每个页面单独堆发光。
5. 改数据库前先备份，不要直接 `drop` 或让 Hibernate 自动改表。
6. 遇到官方资产相关逻辑，默认只读；只有 admin 后台可管理。
7. 每次大改后至少跑 `npm run lint` 和 `npm run build`。
